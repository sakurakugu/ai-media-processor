# Architecture Notes

## 目标

第一阶段只解决一件事：稳定完成本地图片 5 分类。

分类标签：

- `screenshot_text`
- `cosplay`
- `anime_art`
- `meme`
- `other`

## 为什么先做固定分类

- 当前目标是把工具跑起来，不是追求一步到位
- 固定标签更容易做 Prompt 约束和结果校验
- 后续 OCR、截图细分类、梗图子类，都可以在这个骨架上扩展

## 建议模型策略

首选：

- `Qwen3.5-4B`

原因：

- 视觉理解能力足够做第一版
- 对 `16GB VRAM` 更友好
- 速度、显存、可用性之间平衡更合理

备选：

- `Qwen3.5-9B`

适用条件：

- 你确认 4B 的误判较多
- 你能接受更慢
- 你的本地部署方式能稳定承载更大的模型

## 处理流程

### 1. 输入

- 单张图片
- 多张图片
- 一个目录
- GUI 或 CLI 入口

### 2. 预处理

- 过滤扩展名
- 统一图片路径
- 可选缩略图预览

### 3. 模型推理

发送固定 Prompt 到视觉模型，要求只输出 JSON：

```json
{
  "label": "anime_art",
  "confidence": 0.87,
  "reason": "non-photorealistic anime style illustration"
}
```

## Prompt 约束原则

- 只允许输出 5 个标签之一
- 不确定时输出 `other`
- 不要求长解释
- 返回 JSON，便于程序消费

## 当前实现取舍

### 先不做

- OCR
- 梗图子类
- 截图细分类
- 嵌入向量缓存
- 微调

### 现在先做

- GUI
- CLI
- 后端抽象
- 批量调用
- 结果导出

## 风险与应对

### 模型输出不稳定

应对：

- 强约束 Prompt
- 统一 JSON 解析
- 解析失败时回退为 `other`

### 本地部署复杂

应对：

- GUI 与后端解耦
- 先用 `mock` 跑通工具层
- 之后再替换为真实模型

### 分类边界模糊

应对：

- 第一版接受 `other` 占比偏高
- 后续通过人工复核样本补规则

## 第二阶段扩展方向

- 先加 OCR 特征
- 再做二阶段分类
- 加本地缓存和复核面板
- 支持目录监控和增量处理
